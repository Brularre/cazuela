## 2026-04-08 — WhatsApp provider selection

**Suggested:**
Use Twilio over Meta's direct WhatsApp Business API
or unofficial libraries (WhatsApp Web.js).

**Decision:** Accepted

**Rationale:**
Twilio sandbox eliminates Meta business verification wait (days to weeks).
Cost is negligible for personal use (~$0.005/msg).
Unofficial libraries risk account bans — not acceptable for a project meant to be shared.

---

## 2026-04-08 — Database selection

**Suggested:**
Supabase (managed Postgres) over SQLite, MongoDB Atlas, or Railway Postgres.

**Decision:** Accepted

**Rationale:**
Supabase's UI for browsing data is a significant quality-of-life advantage during development.
SQLite is ruled out because it doesn't survive deploys on ephemeral filesystems.
MongoDB's document model adds unnecessary complexity for relational data like expenses and users.

---

## 2026-04-08 — Monorepo vs separate repositories

**Suggested:**
Single monorepo for backend (Python/FastAPI) and frontend (Next.js).

**Decision:** Accepted

**Rationale:**
One developer, one product — separate repos add overhead with no benefit at this scale.
Shared types between backend and frontend are easier to manage.
Splitting later is straightforward; merging is not.

## 2026-04-08 — Deployment model (shared instance vs self-hosted)

**Suggested:**
Hybrid — single hosted instance where users are identified by WhatsApp number,
with users optionally bringing their own Anthropic API key.
Self-hosting documented for technical users.

**Decision:** Accepted

**Rationale:**
Non-technical friends can use a shared instance with zero setup (just save a contact).
Technical users can self-host for full data control.
Broader reach than pure self-hosted model without the complexity of managing per-user infrastructure.

## 2026-04-08 — AI mode vs manual mode

**Suggested:**
Manual mode (regex/keyword parsing) as default.
AI mode (Claude Haiku) unlocked when user provides their own Anthropic API key.

**Decision:** Accepted

**Rationale:**
Keeps the app free to run for users who don't want to pay for AI.
Manual mode also works as a fallback and is faster for structured inputs.
The cost split is fair — users who want AI pay for their own usage.

## 2026-04-08 — Stateful conversation memory

**Suggested:**
Store last ~10 messages per user in DB and pass as context to Claude on each call.
Store long-term preferences in user profile.

**Decision:** Accepted

**Rationale:**
Gives the personal assistant feel at minimal cost (~$0.001 extra per call with Haiku).
The stateless alternative forces every message to be fully self-contained,
which breaks natural conversation flow.

**Decision:** Accepted

## 2026-04-08 — Fixed expense categories with AI mapping

**Suggested:**
Define a fixed category list
(comida, transporte, salud, hogar, entretenimiento, ropa, tecnología, educación, viajes, otros).
Claude maps free-text descriptions to this list.

**Decision:** Accepted over free-form categories

**Rationale:**
Fixed categories make dashboard charts and budget limits consistent and queryable.
Free-form categories make aggregation unreliable.
AI mapping gives the flexibility of natural language while keeping data structured.

## 2026-04-09 — Phase 0 complete: end-to-end WhatsApp loop working

**Suggested:**
Deploy to Railway as the public webhook URL instead of using ngrok.

**Decision:** Accepted

**Rationale:**
Work machine security software blocks all tunneling processes.
Railway gives a permanent public HTTPS URL that Twilio can reach,
and auto-deploys on every push to main.
Verified end-to-end: WhatsApp message → Twilio → Railway webhook → reply received.

## 2026-04-09 — Local tunnel replacement: Railway early deployment

**Suggested:**
Replace ngrok/localtunnel with Railway deployment for Twilio webhook testing.
Bruno is on a work machine with endpoint security software
that kills all tunneling processes (SSH, git-remote-https, localtunnel).

**Decision:** Accepted

**Rationale:**
No tunneling tool can be installed or run on the machine.
Railway deployment (triggered by git push) takes ~30 seconds
and provides a real HTTPS URL for Twilio.
All logic testing (unit + integration) remains fully local.
Only Twilio webhook testing requires a push — a minor tradeoff
that results in a more realistic dev workflow anyway.
