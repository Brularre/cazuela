## 2026-04-08 — WhatsApp provider selection

**Suggested:**
Use Twilio over Meta's direct WhatsApp Business API
or unofficial libraries (WhatsApp Web.js).

**Decision:** Accepted

**Rationale:**
Twilio sandbox eliminates Meta business verification wait (days to weeks).
Cost is negligible for personal use (~$0.005/msg).
Unofficial libraries risk account bans — not acceptable for a project meant to be shared.

---

## 2026-04-08 — Database selection

**Suggested:**
Supabase (managed Postgres) over SQLite, MongoDB Atlas, or Railway Postgres.

**Decision:** Accepted

**Rationale:**
Supabase's UI for browsing data is a significant quality-of-life advantage during development.
SQLite is ruled out because it doesn't survive deploys on ephemeral filesystems.
MongoDB's document model adds unnecessary complexity for relational data like expenses and users.

---

## 2026-04-08 — Monorepo vs separate repositories

**Suggested:**
Single monorepo for backend (Python/FastAPI) and frontend (Next.js).

**Decision:** Accepted

**Rationale:**
One developer, one product — separate repos add overhead with no benefit at this scale.
Shared types between backend and frontend are easier to manage.
Splitting later is straightforward; merging is not.

**Suggested:**
Extract aggregation logic from `get_week_summary` into a pure
`aggregate_by_category` function, tested with a shared fixture file.

**Decision:** Accepted

**Rationale:**
The original function mixed DB querying with aggregation logic,

**Suggested (by Bruno, architecture shaped collaboratively):**
Integrate MCP (Model Context Protocol) as a multi-step
agent flow for ambiguous expenses, satisfying a course assignment
requirement alongside the Week 2 feature additions.

**Decision:** Accepted — implemented with a deterministic stub agent

**Rationale:**
Using a real Claude API call for the MCP agent would cost money

a real model call later with no interface changes.

## 2026-04-15 — MCP confirm/cancel flow: prefix lookup

**Suggested:**
Show the user the first 8 characters of the context UUID
as a short code to confirm or cancel an ambiguous expense.

**Decision:** Accepted — with a prefix scan added to the context store

**Rationale:**
Typing a full UUID in WhatsApp is impractical.
An 8-character hex prefix is short enough to type while still being
unique in practice (collision probability negligible for a personal
assistant with low concurrency).
A `find_by_prefix()` function was added to `context.py`
to support the lookup.

## 2026-04-15 — Twilio signature validation: optional by env var

**Suggested:**
Validate the `X-Twilio-Signature` header on the `/webhook` endpoint
to prevent spoofed requests.

**Decision:** Accepted — validation skipped when `TWILIO_AUTH_TOKEN` is unset

**Rationale:**
Hard-requiring the token would break local tests and dev deploys
where the token isn't configured.
Making it conditional on the env var being set means production
is secure (token is set in Railway) while the dev workflow is unchanged.
The alternative — always validate — would require mocking
Twilio's RequestValidator in every test that hits the webhook.

## 2026-04-16 — MCP removed from direct-write handlers

**Suggested:**
Wrap all handler DB inserts in MCP context tracking
(send_context / confirm / rollback) for rollback support.

**Decision:** Rejected for direct-write handlers;
retained only for the ambiguous expense flow

**Rationale:**
For simple inserts (todos, notes, shopping, wishlist)
the rollback is cosmetic — the DB was never written to
on failure, so there is nothing to undo.
Worse, MCP added two extra DB round-trips per write
and introduced a production bug when status enforcement
(`confirm` required "staged") broke handlers that never
called `request_action`.
MCP earns its keep only where there is a genuine two-message
interaction: propose → user confirms/cancels.

---

## 2026-04-16 — MCP context store: in-memory to Supabase

**Suggested:**
Replace the module-level `_store` dict in `context.py`
with Supabase table operations.

**Decision:** Accepted

**Rationale:**
Railway restarts the container on every deploy,
wiping in-memory state. A user who said "pagué 3000"
and then deployed (or had a server restart) before
replying "confirmar" would silently lose the staged expense.
The Supabase-backed store survives deploys.
Tests were rewritten to use a stateful `FakeClient` fixture
instead of accessing `_store` directly.
